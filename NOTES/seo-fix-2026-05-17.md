## TL;DR

The site code was already 95% correct before this pass. Three JSON-LD `url` fields were missing a trailing slash; those were fixed. **The actual blocker for Google's index-validation status is almost certainly operator-side configuration** (GitHub Pages settings and DNS), not the HTML. See the manual checklist at the end.

## The reported Search Console issues

**Cluster 1 — "Page with redirect" (Validation FAILED, 3 URLs):**
- `https://surfacingapp.com/index.html` (last crawled May 14)
- `http://www.surfacingapp.com/` (last crawled May 9)
- `https://www.surfacingapp.com/` (last crawled May 8)

**Cluster 2 — "Alternate page with proper canonical tag" (Validation STARTED, 3 URLs):**
- `http://surfacingapp.com/`
- `http://surfacingapp.com/download.html`
- `http://surfacingapp.com/index.html`

## Translation from Google-speak

Cluster 2 is actually fine. "Alternate page with proper canonical tag" + "Validation STARTED" means Google has seen these `http://` variants, noticed they carry a canonical pointing to the HTTPS form, and is currently reconciling them. That's the system working as designed. Patience.

Cluster 1 is the louder one. "Page with redirect" + "Validation FAILED" doesn't mean the redirect is broken; it means Google's last validation pass couldn't confirm the URLs were "fixed" because — surprise — they're still redirects, and they're supposed to be. This category tends to persist while Google deduplicates aliases. A re-validation request usually clears it, *provided* the underlying redirects actually work.

Which is where the operator-side checks come in: if `https://www.surfacingapp.com/` doesn't 301 to `https://surfacingapp.com/` because the www DNS or GitHub Pages settings are off, those URLs never become true redirects and Google never resolves them.

## What changed in this pass

Three JSON-LD `url` values that referred to the homepage as `https://surfacingapp.com` (no trailing slash) were normalized to `https://surfacingapp.com/` (with trailing slash) to exactly match `<link rel="canonical" href="https://surfacingapp.com/">`. Google generally treats the two as equivalent, but consistency across canonical + og:url + JSON-LD is a free win and removes one variable from any future debugging session.

Specific edits:

- `index.html` line 32 — WebSite `url` — added trailing slash.
- `index.html` line 46 — SoftwareApplication `url` — added trailing slash.
- `download.html` line 103 — SoftwareApplication `url` — added trailing slash.

No content was touched. No CSS, JS, or layout files were touched. No internal links needed changing — they were already clean.

## What did NOT change (and why)

- **`<link rel="canonical">` tags** — already correct on all 9 pages.
- **The `/index.html` → `/` JS redirect in `index.html`** — already works correctly. It only fires when `location.pathname` ends with `/index.html`, and it preserves query strings and hash fragments via `location.href.replace(...)`. Good as-is.

## What's actually blocking Search Console — manual operator steps

These are things this pass cannot touch from the repo. Verify in order:

**1. GitHub Pages settings** — at `https://github.com/ckmoonunit/surfacing-website/settings/pages`:
   - Custom domain field reads exactly `surfacingapp.com` (no www).
   - "Enforce HTTPS" checkbox is enabled.
   - If "Enforce HTTPS" is greyed out, the cert is still provisioning; wait and revisit.

**2. DNS records** — at the registrar managing `surfacingapp.com`:
   - Apex `surfacingapp.com` has A records pointing at all four GitHub Pages IPs:
     `185.199.108.153`, `185.199.109.153`, `185.199.110.153`, `185.199.111.153`
     (Or, if the registrar supports it, an `ALIAS`/`ANAME` record to `ckmoonunit.github.io`.)
   - `www.surfacingapp.com` has a CNAME pointing at `ckmoonunit.github.io`. Without this, `https://www.surfacingapp.com/` cannot resolve, GitHub cannot redirect it to the apex, and that URL stays stuck in Search Console forever.
   - Test from a fresh terminal:
     ```
     dig +short surfacingapp.com
     dig +short www.surfacingapp.com
     curl -I https://www.surfacingapp.com/        # should see 301 → https://surfacingapp.com/
     curl -I http://surfacingapp.com/             # should see 301 → https://surfacingapp.com/
     curl -I https://surfacingapp.com/index.html  # 200; redirect happens client-side via JS
     ```

**3. Re-trigger Search Console validation** — once 1 and 2 check out:
   - In Search Console → Indexing → Pages → "Page with redirect" → click **Validate Fix**.
   - In Search Console → Indexing → Pages → "Alternate page with proper canonical tag" → leave it alone; "Validation STARTED" resolves on its own.
   - Validation can take days to a week. The status text isn't a real-time check; Google re-crawls a sample, sees the same redirect/canonical setup, and either re-classifies or doesn't.

**4. Optional, helpful** — in Search Console → Sitemaps, re-submit `https://surfacingapp.com/sitemap.xml`. If it's already submitted, hit the "..." menu and pick "Resubmit". Cheap nudge for the crawler.
